# Krok 3.4: Algorytm DFS dla Propagacji Przepływów - Specyfikacja Techniczna

Ten dokument zawiera szczegółowe kroki techniczne do wdrożenia mechanizmu propagacji przepływów powietrza (Flow Propagation) opartego na algorytmie DFS (Przeszukiwanie w głąb), zgodnie z dokumentacją dostarczoną w `03.4-flow-propagation-dfs.md`.

Naszym głównym założeniem jest umożliwienie pełnej, dynamicznej sumacji wydatków powietrza (w m³/h) z terminali (kratek, anemostatów) przez wszystkie przewody aż do korzenia sieci (reprezentowanego przez Centrale Wentylacyjne `AHU`, Wentylatory `FAN` lub węzły sumujące `VIRTUAL_ROOT`).

## 1. Zmiany w Obiektach Danych (Model)

### 1.1 Aktualizacja `src/types.ts`
Do interfejsu rur (`DuctSegment`) należy dodać nowe pole przechowujące aktualny przepływ powietrza. Rury te pełnią w grafie rolę Krawędzi (`DuctEdge`).

```typescript
// W src/types.ts
export interface DuctSegment {
  // ... inne pola
  
  // Aerodynamics (Nowe Pola)
  flowRate?: number; // m³/h - wyliczony wydatek powietrza przypływający przez rurę
}
```

## 2. Implementacja Silnika Przepływowego (`NetworkEngine`)

Ze względu na skomplikowanie logiki grafowej, należy wydzielić ją do osobnego serwisu, zamiast sztucznie pogrubiać kod w `useDuctStore.ts`.

### 2.1 Utworzenie `src/lib/networkEngine.ts`
Plik ten powinien eksportować czystą funkcję `calculateNetworkFlows(nodes, edges)`, przyjmującą obecny stan węzłów i krawędzi, a zwracają kopie obiektów ze zaktualizowanym polem `flow` (dla node'ów) i `flowRate` (dla segmentów).

#### Plan Algorytmu:
1. **Budowa Mapy Sąsiedztwa (Adjacency Map)**: Utwórz `Map<string, string[]>`, gdzie kluczem jest ID węzła, a wartością tablica ID połączonych krawędzi (segmentów).
2. **Identyfikacja Korzeni**: Używając funkcji `Array.filter`, wyłap węzły pełniące funkcję generatorów, tzn. węzły gdzie `componentCategory === 'EQUIPMENT'` lub `componentCategory === 'VIRTUAL_ROOT'`. Ważne: systemy muszą się zgadzać, ale dla uproszczenia (grafy nie mają pętli) zaczynamy obchód od każdego urządzenia.
3. **Rekurencja (DFS)**:
   Funkcja `dfs(currentNodeId, incomingEdgeId)` wraca sumaryczny wydatek z poszczególnych "gałęzi":
   - **Warunek brzegowy**: Dla `componentCategory === 'TERMINAL'`, należy pobrać przypisane do niego pole `zoneId` i pobrać docelowy przepływ (`targetFlow`) ze strefy (bądź od razu z pola `flow` tego node'a jeśli synchronizacja ze strefą działa poprawnie).
   - **Przejście**: Pętla po wszystkich krawędziach węzła (pomijamy `incomingEdgeId`). Na wyjściu `branchFlow = dfs(nextNodeId, currentEdgeId)`.
   - **Zapis**: Do klonu bieżącej krawędzi (`updatedEdges[currentEdgeId]`) przypisywany jest `flowRate = branchFlow`.
   - **Suma**: `totalFlow += branchFlow`.
   - **Zapis węzła**: Zaktualizuj klon wezła (`updatedNodes[currentNodeId]`) uaktualniając `flow = totalFlow`.

## 3. Integracja ze stanem (`useDuctStore.ts`)

W `src/stores/useDuctStore.ts` dodaj wywoływanie akcji `recalculateFlows()`.

```typescript
recalculateFlows: () => {
  const { nodes, edges } = get();
  // W przyszłości przekażemy tu też strefy, żeby odczytywać flow terminali,
  // na razie zakładamy że node.flow przechowuje już flow ze strefy
  
  const { updatedNodes, updatedEdges } = calculateNetworkFlows(nodes, edges);
  set({ nodes: updatedNodes, edges: updatedEdges });
}
```

### 3.1 Wyzwalanie (Triggers)
Funkcja `recalculateFlows` musi operować po absolutnie każdej zmianie, która może przerwać ciągłość grafu lub dopisać do niego wydatek z terminala:
- Po wywołaniu `addNode`, `removeNode`.
- Po wywołaniu `addEdge`, `removeEdge`, i w szczególności `updateEdge`.
- Funkcje takie jak `splitEdge`, `mergeNodeToEdge`, `mergeNodes`, `insertInlineComponent`.

### 3.2 Aktualizacja z poziomu `useZoneStore`
Pamiętaj, by w `useZoneStore.ts` przy operacji na strefie (np. `updateZone`), gdzie zmienia się wymagany przebieg u dołu tabeli m³/h (np. `manualSupply`, `manualExhaust`), upewnić się, że przepływ ten jest poprawnie kopiowany do przypiętych terminali HVAC w tej strefie (w `DuctNode.flow`) a następnie ręcznie zawołać `useDuctStore.getState().recalculateFlows()`. Nasłuchiwanie tego eventu zapewni synchronizację real-time w miarę zmian w widoku tabelarycznym `Air Balance`.
