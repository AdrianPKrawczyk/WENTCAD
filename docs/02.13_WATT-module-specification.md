# **Dokumentacja Techniczna: WENTCAD Architecture & Thermal Topology (WATT) \- v2.0**

## **1\. Cel i Zakres Modułu**

Moduł WATT służy do automatycznej transformacji geometrii 2D (strefy i obrysy) w analityczny model cieplny. Kluczowym celem jest zidentyfikowanie typu każdej krawędzi pomieszczenia, przypisanie jej fizycznych właściwości materiałowych oraz automatyczny import stolarki otworowej z plików DXF.

## ---

## **2\. Model Danych i Klasy Przegród (Zgodność ze Standardem IFC)**

### **2.0. Założenia Architektoniczne**

Wewnętrzny model danych aplikacji WENTCAD w obszarze architektury i fizyki budowli został zaprojektowany w oparciu o hierarchię klas **IFC (Industry Foundation Classes \- ISO 16739\)**.

Przyjęcie tego standardu na poziomie struktury stanu (Zustand) gwarantuje:

1. Bezstratny eksport analityczny do formatów otwartych (IFC2x3, gbXML).  
2. Gotowość struktury danych do obsługi rygorystycznych algorytmów obliczeniowych OZC (zapotrzebowanie na moc cieplną i chłodniczą).  
3. Uniknięcie redundancji danych poprzez zastosowanie wzorca projektowego *Typ-Instancja* (Type-Instance pattern).

---

### **2.1. Hierarchia Definicji Materiałowych**

Fizyka przegród budowlanych w systemie jest modelowana warstwowo (od najprostszego surowca po kompletną "kanapkę" budowlaną). W globalnym stanie projektu (useProjectStore) zdefiniowane są następujące encje:

#### **2.1.1. IfcMaterial (Baza surowcowa)**

Podstawowa jednostka definiująca izotropowy materiał fizyczny. Nie posiada wymiarów geometrycznych.

* **Właściwości cieplne (wymagane do obliczeń statycznych i dynamicznych):**  
  * ThermalConductivity ($\\lambda$) $\[W/(m \\cdot K)\]$: Współczynnik przewodzenia ciepła.  
  * MassDensity ($\\rho$) $\[kg/m^3\]$: Gęstość objętościowa.  
  * SpecificHeatCapacity ($c\_p$) $\[J/(kg \\cdot K)\]$: Ciepło właściwe (kluczowe do obliczania pojemności cieplnej budynku w lecie).  
* *Przykłady:* Beton zbrojony, Styropian EPS, Wełna mineralna.

#### **2.1.2. IfcMaterialLayer (Warstwa materiałowa)**

Definiuje użycie konkretnego materiału w przegrodzie o określonej grubości.

* **Zależności:** Posiada referencję (ID) do jednego obiektu IfcMaterial.  
* **Parametry:**  
  * LayerThickness ($d$) $\[m\]$: Grubość warstwy.  
* **Obliczenia pochodne:** System na tym poziomie wylicza opór cieplny warstwy: $R\_i \= d / \\lambda$.

#### **2.1.3. IfcMaterialLayerSet (Zestaw warstw / Konstrukcja)**

Kompletna definicja układu warstw tworzących fizyczną przegrodę. Odpowiada znacznikowi \<Construction\> w standardzie gbXML.

* **Zależności:** Posiada uporządkowaną tablicę obiektów IfcMaterialLayer (kolejność ma znaczenie dla kierunku przepływu ciepła i akumulacji).  
* **Obliczenia pochodne:** \* Całkowity opór cieplny: $R\_{tot} \= R\_{si} \+ \\sum R\_i \+ R\_{se}$  
  * Współczynnik przenikania ciepła: $U \= 1 / R\_{tot} \\ \[W/(m^2 \\cdot K)\]$

---

### **2.2. Typologia Elementów Budowlanych (Type Objects)**

Aby powiązać abstrakcyjną definicję materiałową (IfcMaterialLayerSet) z logiczną funkcją przegrody w budynku, system wykorzystuje klasy typologiczne.

#### **2.2.1. IfcWallType / IfcSlabType / IfcRoofType**

Definicje powtarzalnych elementów budowlanych w katalogu projektu.

* **Zależności:** Każdy typ posiada referencję do jednego IfcMaterialLayerSet.  
* **Parametry klasyfikacyjne:**  
  * PredefinedType: Enum określający funkcję (np. dla ściany: SOLIDWALL, PARTITIONING; dla stropu: FLOOR, ROOF).  
  * IsExternal: Wartość boolean definiująca kontakt ze środowiskiem zewnętrznym.

*Implementacja w WENTCAD:* Silnik topologiczny WATT, po wykryciu odsunięcia między poligonem strefy a obrysem zewnętrznym, tworzy krawędź (Boundary) i automatycznie mapuje na nią odpowiedni IfcWallType oznaczony jako IsExternal, którego sumaryczna grubość warstw ($\\sum d$) odpowiada wykrytemu odsunięciu.

---

### **2.3. Komponenty Otworowe i Transparentne (Stolarka)**

Otwory w przegrodach podlegają odrębnej klasyfikacji ze względu na specyfikę przenoszenia promieniowania słonecznego.

#### **2.3.1. IfcWindowStyle / IfcDoorStyle (Katalog Stolarki)**

Definicja powtarzalnych typów okien i drzwi.

* **Parametry cieplne i optyczne:**  
  * OverallUValue ($U\_w$) $\[W/(m^2 \\cdot K)\]$ \- Całkowity współczynnik przenikania okna.  
  * SolarHeatGainCoefficient ($g$) $\[0-1\]$ \- Całkowita przepuszczalność energii promieniowania słonecznego. Parametr krytyczny dla obliczeń zysków chłodniczych.  
  * VisibleLightTransmittance ($T\_v$) $\[0-1\]$ \- Opcjonalnie do analizy oświetlenia naturalnego.  
* **Wymiary bazowe:**  
  * Szerokość ($B$) i Wysokość ($H$). Z poziomu parsera DXF parametry te są weryfikowane z nazwami warstw (np. OKNO\_H150).

#### **2.3.2. Instancje Otworów (IfcWindow / IfcDoor)**

Rzeczywiste obiekty wstawione na rzut w module Workspace2D.

* **Zależności przestrzenne:** Zawsze przypisane do konkretnej ściany (krawędzi poligonu w useZoneStore), wykorzystując relację typu IfcRelFillsElement.  
* **Parametry geometryczne instancji:**  
  * Placement: Wektor pozycji na długości ściany $L$.  
  * SillHeight ($H\_o$): Wysokość parapetu nad poziomem podłogi.

---

### **2.4. Struktura w stanie TypeScript (Zustand Abstraction)**

Dla zapewnienia wysokiej wydajności w środowisku webowym (React), pełen schemat IFC jest spłaszczony do znormalizowanych struktur relacyjnych.

**Szkielet encji w useProjectStore (Słowniki/Katalogi):**

TypeScript

interface LibraryStore {

  materials: Record\<string, IfcMaterial\>;

  layerSets: Record\<string, IfcMaterialLayerSet\>;

  wallTypes: Record\<string, IfcWallType\>;

  windowTypes: Record\<string, IfcWindowStyle\>;

}

**Szkielet encji w useZoneStore (Instancje na rzucie 2D):**

TypeScript

interface ZoneBoundary {

  id: string;

  relatedWallTypeId: string; // Wskaźnik do katalogu IfcWallType

  isExternal: boolean;

  geometry: {

    lengthNet: number; // L\_osi

    azimuth: number; // Kąt od Północy (do zysków słonecznych)

  };

  openings: Array\<{

    id: string;

    windowTypeId: string; // Wskaźnik do katalogu IfcWindowStyle

    positionOffset: number;

  }\>;

}

Powyższa architektura danych ściśle izoluje definicje materiałowe (fizykę) od geometrii (instancji), co jest podstawowym wymogiem dla wydajnego działania modułu OZC oraz generatorów plików gbXmlExport i ifcExport.

## **3\. Strategia Importu DXF (Automatyzacja Geometrii i Metadanych)**

### **3.0. Założenia Systemowe**

Moduł importu podkładów DXF (zintegrowany w komponencie DxfOutlinesModal.tsx oraz obsługiwany przez bibliotekę dxfUtils.ts) pełni rolę głównego interfejsu wprowadzania danych geometrycznych. Zamiast wymuszać na użytkowniku ręczne rysowanie modelu BIM, system WENTCAD wykorzystuje koncepcję **Smart Mappingu** (inteligentnego mapowania warstw), transformując płaskie, dwuwymiarowe obiekty CAD w semantyczne encje modelu cieplnego.

---

### **3.1. Kategoryzacja i Mapowanie Warstw (Layer Mapping)**

Podczas importu pliku DXF, interfejs użytkownika wymusza przypisanie wybranych warstw z pliku CAD do jednej z czterech rygorystycznych ról topologicznych. Tylko encje znajdujące się na zmapowanych warstwach podlegają dalszemu procesowaniu.

* **Warstwa Stref (Thermal Zones) \- wykorzystanie istniejącej funkcji improtu obrysów do stref bilansu wentylacyjnego:**  
  * *Oczekiwana geometria:* Zamknięte polilinie (LWPOLYLINE).  
  * *Cel:* Generowanie instancji pomieszczeń w useZoneStore. Polilinia wyznacza wewnętrzny obrys wymiany ciepła.  
* **Warstwa Obrysu Budynku (Building Footprint \- Outer Ring):**  
  * *Oczekiwana geometria:* Zamknięta polilinia obejmująca cały budynek.  
  * *Cel:* Generowanie zewnętrznej granicy ewaluacji (referencja do obliczania grubości ścian zewnętrznych i detekcji środowiska). Zapisywana w useProjectStore.  
* **Warstwa Dziedzińców (Courtyards \- Inner Rings) \[Opcjonalna\]:**  
  * *Oczekiwana geometria:* Zamknięte polilinie wewnątrz Obrysu Budynku.  
  * *Cel:* Wycinanie "otworów" w globalnym obrysie. Umożliwia poprawne rozpoznanie ścian zewnętrznych wychodzących na patio/dziedziniec.  
* **Warstwy Stolarki Otworowej (Smart Windows & Doors):**  
  * *Oczekiwana geometria:* Zamknięte polilinie o 4 wierzchołkach (prostokąty).  
  * *Cel:* Generowanie obiektów typu OpeningCandidate, które w kolejnym kroku zostaną geometrycznie "przyklejone" do ścian.

---

### **3.2. Ekstrakcja Geometrii Bazowej (Geometry Parsing)**

Po zatwierdzeniu mapowania warstw, parser dxfUtils.ts wykonuje ekstrakcję wektorów. Z uwagi na specyfikę stolarki otworowej, system stosuje dedykowaną logikę do odczytu wymiarów okien z prostokątów narysowanych na rzucie.

* **Ekstrakcja szerokości okna ($B$):**  
  System bada cztery boki prostokąta reprezentującego okno. Dłuższy bok prostokąta jest interpretowany jako szerokość otworu w rzucie ($B$), natomiast krótszy bok (często odpowiadający grubości ściany na rysunku CAD) jest ignorowany analitycznie.  
* **Środek ciężkości (Centroid):**  
  Dla każdego zidentyfikowanego prostokąta okna wyliczany jest jego środek ciężkości. Zmienna ta posłuży silnikowi topologicznemu (Spatial Join) do określenia, do której krawędzi pomieszczenia należy "przykleić" dany otwór.

---

### **3.3. Inteligentne Metadane z Nazw Warstw (Regex Metadata Parsing)**

Aby wyeliminować konieczność ręcznego wprowadzania wysokości okien, system WENTCAD wdraża mechanizm pozyskiwania parametrów Z (wysokościowych) bezpośrednio z łańcucha znaków stanowiącego nazwę warstwy DXF.

* **Wzorzec Wyrażenia Regularnego (Regex):**  
  System analizuje nazwy warstw stolarki przy użyciu następującego wzorca: ^.\*\_H(\\d+)(?:\_Ho(\\d+))?.\*$  
* **Reguły Mapowania:**  
  * Zmienna H (Grupa 1): Mapowana na wysokość całkowitą okna/drzwi w centymetrach.  
  * Zmienna Ho (Grupa 2 \- Opcjonalna): Mapowana na wysokość parapetu (Sill Height) w centymetrach.  
* **Przykłady przetwarzania przez parser:**  
  * ARCH\_OKNA\_H150\_Ho90 $\\rightarrow$ Wysokość okna: 1.5 m, Parapet: 0.9 m. Typ: Okno.  
  * STOLARKA\_ZEW\_H210\_Ho0 $\\rightarrow$ Wysokość okna: 2.1 m, Parapet: 0.0 m. Typ: Drzwi balkonowe.  
* **Wartości domyślne (Fallback):**  
  W przypadku braku dopasowania Regex (np. warstwa nazywa się po prostu A-GLAZ), system przypisuje obiektom parametry domyślne, pobierane z globalnej konfiguracji piętra (useSettingsStore.defaultWindowHeight oraz useSettingsStore.defaultSillHeight).

---

### **3.4. Transformacja do Stanu Aplikacji (State Hydration)**

Zakończenie importu nie powoduje natychmiastowego wyliczenia bilansu. Zamiast tego, struktury danych są "wlewane" do store'ów Zustand, przygotowując środowisko dla silnika topologicznego (WATT Engine):

1. Polilinie stref tworzą nowe obiekty w useZoneStore.zones (bez przypisanych jeszcze właściwości zewnętrznych/wewnętrznych).  
2. Obrys główny i dziedzińce zapisywane są jako wielokąty w useProjectStore.buildingFootprint.  
3. Wyodrębnione prostokąty stolarki (wraz ze zdekodowanymi parametrami $H$ i $H\_o$) trafiają do tymczasowej tablicy OpeningCandidates, oczekując na proces przypisania (Snapping) do konkretnych krawędzi przegród w fazie analizy architektonicznej.

## **4\. Hybrydowy Silnik Topologiczny (WATT Topology Engine)**

### **4.0. Założenia Systemowe i Architektura Silnika**

Silnik topologiczny WATT (WENTCAD Architecture & Thermal Topology) odpowiada za transformację luźnego zbioru niezależnych wielokątów (poligonów stref i obrysów) w spójny, relacyjny graf 2.5D.

W przeciwieństwie do uproszczonych systemów bazujących wyłącznie na obrysach zewnętrznych, WATT wdraża **algorytm hybrydowy**, który w pierwszej kolejności rozwiązuje relacje wewnątrzbudynkowe (sąsiedztwo stref), a dopiero w drugiej weryfikuje warunki brzegowe (środowisko zewnętrzne). Takie podejście gwarantuje poprawną detekcję dziedzińców, szachtów instalacyjnych oraz ścian wewnętrznych o zróżnicowanych temperaturach.

---

### **4.1. Faza I: Analiza Sąsiedztwa (Internal Adjacency Check)**

Celem pierwszej fazy jest wykrycie wszystkich przegród wewnętrznych oraz powiązanie ze sobą stref graniczących, co jest krytyczne dla obliczania przepływów ciepła między pomieszczeniami o różnej temperaturze projektowej (np. Biuro $20^\\circ C$ a Korytarz $15^\\circ C$).

**Przebieg algorytmu:**

1. **Iteracja po krawędziach:** Dla każdego pomieszczenia (Strefa $A$), system dekomponuje poligon na zbiór odcinków (krawędzi $E\_{A1}, E\_{A2}, ... E\_{An}$).  
2. **Filtrowanie przestrzenne (Spatial Bounding Box):** Aby zoptymalizować wydajność, program odrzuca strefy znajdujące się w odległości większej niż maksymalna dopuszczalna grubość ściany wewnętrznej ($d\_{max\\\_int} \= 0.6 \\ m$).  
3. **Weryfikacja Równoległości:** Krawędź $E\_A$ jest porównywana z krawędziami sąsiedniej Strefy $B$ ($E\_B$). System oblicza kąt między wektorami krawędzi. Jeśli krawędzie są równoległe w ramach tolerancji błędu ($\\Delta \\alpha \\le \\pm 2^\\circ$), przechodzą do kolejnego kroku.  
4. **Rzutowanie Ortogonalne (Overlap Detection):** Odcinki są rzutowane na wspólną oś równoległą. System weryfikuje, czy odcinki "leżą naprzeciwko siebie" (część wspólna rzutów $\> 0$).  
5. **Dystans i Deklaracja (Instantiation):**  
   * Obliczana jest odległość prostopadła między $E\_A$ a $E\_B$ ($d$).  
   * Jeśli $d \\le d\_{max\\\_int}$, obie krawędzie otrzymują typ **INTERIOR**.  
   * Zmienna $d$ zostaje zapisana jako grubość ściany wewnętrznej ($d\_{int}$).  
   * W strukturze Zustand (useZoneStore) zapisywana jest referencja krzyżowa: Strefa $A$ otrzymuje adjacentZoneId: "ID\_Strefy\_B".

---

### **4.2. Faza II: Weryfikacja Brzegowa (External Boundary Check)**

Krawędzie, które po zakończeniu Fazy I nie znalazły żadnego "sąsiada" wewnątrz budynku, przechodzą do weryfikacji ze środowiskiem zewnętrznym.

**Przebieg algorytmu:**

1. **Wejście:** Zbiór krawędzi oznaczonych statusem UNRESOLVED (nieprzypisanych).  
2. **Porównanie z Obrysem Głównym:** Krawędzie są porównywane z wektorami poligonu BuildingFootprint (zarówno z OuterRing \- głównym obrysem, jak i InnerRings \- dziedzińcami/otworami w bryle).  
3. **Aplikacja logiki Fazy I:** System ponownie stosuje testy równoległości ($\\pm 2^\\circ$) oraz rzutowania ortogonalnego względem linii obrysu zewnętrznego.  
4. **Dystans Zewnętrzny:** Mierzona jest odległość prostopadła od krawędzi strefy do krawędzi obrysu DXF.  
   * Wartość ta staje się **rzeczywistą grubością ściany zewnętrznej ($d\_{ext}$)**.  
   * Krawędź otrzymuje typ **EXTERIOR**.  
5. **Obsługa Anomalii (Fallback Logic):** Jeśli krawędź UNRESOLVED nie jest równoległa do obrysu budynku (np. z powodu nieprecyzyjnego narysowania DXF przez architekta) lub odległość jest absurdalnie duża, system domyślnie oznacza ją jako **EXTERIOR** z flagą ostrzegawczą (isAnomalous: true). Jest to bezpieczne założenie (Fail-Safe) w fizyce budowli, zapobiegające niedoszacowaniu strat ciepła.

---

### **4.3. Faza III: Normalizacja Geometrii do Osi (Axis Length Calculation)**

Po sklasyfikowaniu wszystkich krawędzi i wyznaczeniu grubości murów ($d\_{int}$, $d\_{ext}$), silnik modyfikuje surową długość krawędzi z rzutu ($L\_{wew}$), aby uzyskać długość obliczeniową w osiach ($L\_{osi}$) zgodną z normami OZC (np. PN-EN 12831).

Zasada opiera się na analizie dwóch wierzchołków (węzłów topologicznych) kończących dany odcinek ściany zewnętrznej.

Dla każdej krawędzi EXTERIOR:

1. **Węzeł Startowy ($V\_1$) i Węzeł Końcowy ($V\_2$):** System sprawdza, jakie inne krawędzie "spotykają się" w danym wierzchołku strefy.  
2. **Reguły przyrostu ($\\Delta V$):**  
   * **Przypadek A (Narożnik Zewnętrzny):** Jeśli w węźle $V$ krawędź EXTERIOR łączy się z inną krawędzią EXTERIOR, do długości doliczana jest pełna grubość wykrytej ściany zewnętrznej: $\\Delta V \= d\_{ext}$.  
   * **Przypadek B (Styk ze Ścianą Wewnętrzną):** Jeśli w węźle $V$ krawędź łączy się z krawędzią INTERIOR, do długości doliczana jest połowa grubości przyległej ściany wewnętrznej: $\\Delta V \= 0.5 \\cdot d\_{int}$.  
3. **Równanie Końcowe:**  
   $L\_{osi} \= L\_{wew} \+ \\Delta V\_1 \+ \\Delta V\_2$

Uzyskana wartość $L\_{osi}$ nadpisuje długość geometryczną i stanowi od tego momentu jedyną podstawę do obliczania pola powierzchni ściany netto i brutto ($A$).

---

### **4.4. Faza IV: Azymut i Orientacja Geograficzna**

Ostatnim krokiem silnika na płaszczyźnie 2D jest wyznaczenie wektora normalnego ściany w celu obliczenia jej Azymutu (niezbędnego do zysków z promieniowania słonecznego).

1. **Wektor Krawędzi:** Na podstawie znormalizowanych współrzędnych $V\_1$ i $V\_2$ wyznaczany jest wektor krawędzi $\\vec{e}$.  
2. **Wektor Normalny:** Wyznaczany jest prostopadły wektor normalny $\\vec{n}$, zawsze skierowany "na zewnątrz" poligonu strefy (zgodnie z regułą prawej dłoni / kolejnością wierzchołków).  
3. **Korekta Północy:** System pobiera z useProjectStore zmienną NorthAngle (deklinację Północy ustawioną przez użytkownika na rzucie w stopniach).  
4. **Zapis Azymutu:** Kąt wektora normalnego $\\vec{n}$ jest modyfikowany o NorthAngle i zapisywany w instancji krawędzi (np. $180^\\circ$ dla elewacji idealnie południowej). Zmienna ta jest następnie automatycznie dziedziczona przez wszystkie instancje okien (IfcWindow) przypisane do tej ściany.

---

### **4.5. Rezultat Operacji (Data Output)**

Po zakończeniu cyklu WATT, obiekt Zone w Zustand posiada kompletną, przeliczoną strukturę granic, gotową do zasilenia Silnika Obliczeń Cieplnych i weryfikowaną pod kątem eksportu do gbXML bez ryzyka błędu "NullReference". Od tego momentu ingerencja użytkownika ogranicza się wyłącznie do nadpisywania wyliczonych parametrów w sytuacjach nietypowych (np. manualna zmiana typu przegrody z katalogu).

Oto szczegółowe rozwinięcie piątego rozdziału, stanowiące rygorystyczną specyfikację algorytmiczną dla operacji przestrzennych w osi Z. Tekst został sformatowany zgodnie z konwencją dokumentacji technicznej oprogramowania.

---

## **5\. Analiza Pionowa (Multi-Floor Topology Logic)**

### **5.0. Założenia Systemowe i Architektura Silnika**

Analiza pionowa jest modułem silnika geometrycznego WATT, który transformuje zbiór płaskich rzutów (2D) w topologiczny model analityczny przestrzeni (2.5D/3D).

Jej głównym zadaniem jest wyeliminowanie błędu "nieskończonego rzutu" – domyślnego założenia, że każda strefa jest prostopadłościanem przykrytym z góry i z dołu środowiskiem wewnętrznym o tej samej temperaturze. Algorytm wykorzystuje zaawansowane operacje boolowskie na poligonach (Polygon Clipping / Boolean Difference), aby wykryć stropy narażone na kontakt ze środowiskiem zewnętrznym (nawisy, przejazdy, tarasy i dachy szczątkowe).

---

### **5.1. Reprezentacja Danych i Warunki Początkowe**

Zanim silnik przystąpi do obliczeń w osi Z, system musi spełnić określone warunki wstępne dotyczące synchronizacji pięter:

1. **Globalny Układ Współrzędnych:** Poligony stref dla wszystkich kondygnacji muszą operować w ujednoliconym układzie współrzędnych (Global Coordinate System). Rozwiązuje to moduł synchronizacji punktów bazowych dla plików DXF w WENTCAD.  
2. **Struktura Hierarchiczna Kondygnacji:** Z useProjectStore pobierana jest uporządkowana tablica kondygnacji posortowana według rzędnej wysokościowej (Level Elevation) rosnąco: $L\_0, L\_1, \\dots, L\_k$.  
3. **Agregacja Poligonów (Unioning):** Dla każdej kondygnacji $N$ system generuje tymczasowy poligon agregujący ($F\_N$), będący sumą (Boolean Union) wszystkich poligonów stref ogrzewanych na danym piętrze.

---

### **5.2. Algorytm Rzutowania Poligonów (Vertical Boolean Subtraction)**

Sercem modułu jest deterministyczny proces analizy warstwowej dla każdej strefy $S$ znajdującej się na kondygnacji badanej $N$. Poligon strefy $P(S\_N)$ poddawany jest dwóm podstawowym testom: testowi stropu górnego (Roof Test) oraz testowi stropu dolnego (Exposed Floor Test).

#### **5.2.1. Detekcja Przegród Górnych (Dachy i Tarasy)**

Algorytm sprawdza, jaka część strefy $S\_N$ nie jest fizycznie "przykryta" kondygnacją wyższą $N+1$.

1. **Działanie Matematyczne:** Różnica boolowska (Boolean Difference).  
   $R\_{roof} \= P(S\_N) \- F\_{N+1}$  
2. **Ewaluacja Wyniku:**  
   * Jeśli wynikiem jest zbiór pusty ($\\emptyset$), cała strefa $S\_N$ jest przykryta inną kondygnacją. Generowana jest jedna przegroda pozioma typu CEILING\_INTERIOR o powierzchni równej polu strefy.  
   * Jeśli wynikiem jest poligon $R\_{roof} \\neq \\emptyset$, reprezentuje on odsłonięty fragment sufitu. System generuje przegrodę typu **ROOF** (Dach/Taras) o polu $A \= Area(R\_{roof})$. Reszta sufitu (jeśli istnieje) otrzymuje typ CEILING\_INTERIOR.  
3. **Wyjątek (Ostatnia Kondygnacja):** Jeśli kondygnacja $N$ jest najwyższa w hierarchii ($N \= k$), pomija się operację różnicy i natychmiast przypisuje typ **ROOF** dla 100% powierzchni sufitu wszystkich stref na tym piętrze.

#### **5.2.2. Detekcja Przegród Dolnych (Nawisy i Stropy nad Przejazdem)**

Algorytm sprawdza, jaka część strefy $S\_N$ wykracza poza obrys piętra znajdującego się poniżej ($N-1$). Zjawisko to występuje w architekturze przy nadwieszeniach (tzw. wykuszach) oraz pomieszczeniach zaprojektowanych nad przejazdami bramowymi.

1. **Działanie Matematyczne:** Różnica boolowska (Boolean Difference).  
   $R\_{overhang} \= P(S\_N) \- F\_{N-1}$  
2. **Ewaluacja Wyniku:**  
   * Jeśli $R\_{overhang} \= \\emptyset$, podłoga strefy opiera się w pełni na kondygnacji niższej (typ FLOOR\_INTERIOR).  
   * Jeśli $R\_{overhang} \\neq \\emptyset$, zidentyfikowano nadwieszenie. System generuje przegrodę podłogową typu **FLOOR\_EXTERIOR** o polu $A \= Area(R\_{overhang})$.  
3. **Wyjątek brzegowy (Parter / Poziom 0):** Dla $N \= 0$, nie istnieje kondygnacja $N-1$. Cała podłoga otrzymuje typ domyślny **FLOOR\_GROUND** (Podłoga na gruncie), chyba że użytkownik globalnie zdefiniował specyficzny warunek brzegowy (np. budynek na palach).

---

### **5.3. Propagacja Danych i Hydratacja Stanu (State Hydration)**

Po wyliczeniu macierzy różnic boolowskich, otrzymane poligony poddawane są kalkulacji pola powierzchni (za pomocą algorytmu Shoelace / Wzoru Gaussa dla wielokątów).

Uzyskane wartości zasilają zaktualizowaną strukturę HorizontalBoundaries w store'rze strefy (useZoneStore).

**Przykład struktury wyjściowej po analizie:**

TypeScript

    area: 12.5, // m2 (opiera się na parterze)

  },

  {

### **5.4. Wpływ na Obliczenia Cieplne (Thermal Integration)**

Zidentyfikowanie poligonów ROOF oraz FLOOR\_EXTERIOR ma kluczowe znaczenie dla precyzji algorytmu OZC i wpływa na dwie główne składowe bilansu:

1. **Zyski od promieniowania (Sol-Air Temperature / Radiacja):**  
   Obiekty oznaczone jako ROOF poddawane są rygorystycznej ocenie nasłonecznienia. Do obliczeń zysków ciepła latem wykorzystywana jest zastępcza temperatura słoneczna, która dla poziomych przegród górnych przyjmuje wartości krytyczne (znacząco przewyższające temperatury powietrza suchego). Zmienna absorption (współczynnik absorpcji promieniowania, np. jasny/ciemny dach) jest dołączana do modelu ROOF.  
2. **Straty/Zyski przez przenikanie (Conduction):**  
   Powierzchnie FLOOR\_EXTERIOR (stropy nad przejazdem) są traktowane jako pełnoprawne przegrody zewnętrzne o dużej rozpiętości temperatur $\\Delta T$ ($T\_{wew} \- T\_{zew}$), co drastycznie zmienia zapotrzebowanie na moc grzewczą strefy w stosunku do zwykłych stropów międzykondygnacyjnych.

Oto szczegółowe rozwinięcie szóstego rozdziału. Zachowuje ono rygorystyczny, inżynieryjny styl dokumentacji technicznej, stanowiąc architektoniczny wstęp do budowy wbudowanego w WENTCAD silnika obliczeń fizyki budowli.

## **6\. Moduł Obliczeń Cieplnych (WENTCAD Thermal Engine)**

### **6.0. Założenia Systemowe i Cel Modułu**

Moduł Obliczeń Cieplnych stanowi docelową warstwę analityczną aplikacji WENTCAD, konsumującą ustrukturyzowane dane wygenerowane przez silnik topologiczny WATT (Opisany w rozdz. 4 i 5). Wykorzystując znormalizowane wymiary w osiach ($L\_{osi}$), wyodrębnione powierzchnie przegród zewnętrznych, parametry okien oraz topologię pionową, silnik przeprowadza deterministyczne obliczenia zapotrzebowania na moc cieplną (OZC \- projektowanie zimowe) oraz chłodniczą (projektowanie letnie).

Architektura silnika opiera się na normach europejskich (w tym PN-EN 12831 dla strat ciepła) oraz standardach ASHRAE (dla dynamicznych zysków chłodniczych).

Zarys funkcjonalny modulu:

1. **Zyski od promieniowania (Solar Gains):**  
   * Wyznaczenie azymutu okna na podstawie kąta ściany i wektora Północy.  
   * Obliczenie zysków przez oszklenie (uwzględniając $g$-value i zacienienie).  
2. **Przenikanie (Conduction):**  
   * Obliczenie strat/zysków przez ściany, okna, dachy i stropy nad przejazdami ($Q \= U \\cdot A \\cdot \\Delta T$).  
3. **Zyski Wewnętrzne (Internal Loads):**  
   * Pobranie danych o ludziach, oświetleniu i urządzeniach przypisanych do typu pomieszczenia (ObjectType).  
4. **Bilans dynamiczny / statyczny:**  
   * Wyznaczenie szczytowego zapotrzebowania na moc chłodniczą (lato) i grzewczą (zima) dla każdej strefy.

---

### **6.1. Składowa I: Wymiana Ciepła przez Przenikanie (Transmission Heat Transfer)**

Obliczenia przenikania bazują na wyselekcjonowanych krawędziach typu EXTERIOR, ROOF, FLOOR\_EXTERIOR oraz FLOOR\_GROUND.

1. **Równanie bazowe:** $Q\_{tr} \= \\sum (U\_i \\cdot A\_i \\cdot b\_{u,i}) \\cdot (T\_{int} \- T\_{ext})$  
2. **Powierzchnie Netto ($A\_i$):**  
   * Do obliczeń dla ścian zewnętrznych system bezwzględnie wykorzystuje powierzchnię netto: $A\_{netto} \= (L\_{osi} \\cdot H\_{kond}) \- \\sum A\_{openings}$.  
   * Użycie długości $L\_{osi}$ (wyliczonej w Fazie III silnika WATT) gwarantuje uwzględnienie wpływu mostków cieplnych w narożnikach.  
3. **Współczynnik redukcji temperaturowej ($b\_u$):**  
   * Dla przegród stykających się bezpośrednio z powietrzem zewnętrznym (ściany, dachy, stropy nad przejazdem): $b\_u \= 1.0$.  
   * Dla krawędzi typu INTERIOR stykających się ze strefami nieogrzewanymi lub o innej temperaturze (np. klatki schodowe), system automatycznie oblicza $\\Delta T$ na podstawie danych z referencji adjacentZoneId.

---

### **6.2. Składowa II: Zyski Ciepła od Promieniowania Słonecznego (Solar Heat Gains)**

Zyski słoneczne stanowią dominujący czynnik obciążenia chłodniczego w cyklu letnim. Silnik dokonuje podziału na zyski przez przegrody przezroczyste (Stolarka) oraz nieprzezroczyste (Dachy i Ściany).

#### **6.2.1. Przegrody Transparentne (IfcWindow)**

System iteruje przez wszystkie instancje okien przypisane do krawędzi EXTERIOR.

1. **Azymut i Kąt Padania:** Na podstawie wektora normalnego ściany (z poprawką na globalny NorthAngle), silnik ustala orientację geograficzną okna (np. $S, SE, W$).  
2. **Równanie Zysków:** $Q\_{sol,w} \= A\_w \\cdot I\_{rad} \\cdot g \\cdot Z$  
   * $A\_w$: Powierzchnia okna zidentyfikowana z obrysu DXF.  
   * $I\_{rad}$: Natężenie promieniowania słonecznego $\[W/m^2\]$ dla danego azymutu i szerokości geograficznej (baza danych klimatycznych).  
   * $g$: Całkowita przepuszczalność energii słonecznej okna (pobrana ze słownika IfcWindowStyle).  
   * $Z$: Współczynnik zacienienia.

#### **6.2.2. Przegrody Nieprzezroczyste (Nasłonecznienie Dachów)**

Wyodrębnione w analizie pionowej (Rozdz. 5\) przegrody typu ROOF poddawane są analizie z wykorzystaniem **Temperatury Słonecznej (Sol-Air Temperature)**.

Ze względu na absorpcję promieniowania (uzależnioną od koloru elewacji/dachu), nagrzane dachy przekazują ciepło do wnętrza strefy z opóźnieniem fazowym (przesunięcie pojemnościowe). Moduł oblicza to obciążenie na podstawie właściwości materiałowych zdefiniowanych w IfcMaterialLayerSet (Gęstość $\\rho$, Ciepło właściwe $c\_p$).

---

### **6.3. Składowa III: Wewnętrzne Zyski Ciepła (Internal Heat Loads)**

Obciążenie cieplne pochodzące od użytkowników i urządzeń jest generowane na podstawie metadanych przypisanych do strefy w useZoneStore (właściwość ObjectType, np. OFFICE, CONFERENCE\_ROOM, CORRIDOR).

1. **Zyski od Ludzi ($Q\_{occ}$):**  
   * Generowane w oparciu o wskaźnik powierzchniowy (np. $1 \\ osoba / 10 \\ m^2$) lub dokładną liczbę osób (zmienna peopleNumber).  
   * Rozbicie na ciepło jawne (Sensible) i utajone (Latent) – kluczowe dla doboru odpowiedniej wielkości chłodnic i osuszaczy w module AHU (Centrali Wentylacyjnej).  
2. **Zyski od Oświetlenia i Wyposażenia ($Q\_{light}, Q\_{eq}$):**  
   * Posiłkując się polem powierzchni strefy ($Area$), silnik oblicza moc zysków: $Q\_{light} \= Area \\cdot UnitPower\_{light} \\ \[W/m^2\]$.

---

### **6.4. Składowa IV: Wymiana Ciepła przez Wentylację (Ventilation Loads)**

Ponieważ WENTCAD jest systemem do projektowania HVAC, bilans strefy musi uwzględniać strumienie powietrza narzucone przez sieć kanałów (Duct Network).

1. **Równanie dla ciepła jawnego:** $Q\_{v} \= \\dot{V}\_{su} \\cdot \\rho \\cdot c\_p \\cdot (T\_{int} \- T\_{su})$  
   * $\\dot{V}\_{su}$: Strumień objętościowy powietrza nawiewanego do strefy $\[m^3/h\]$.  
   * $T\_{su}$: Temperatura powietrza nawiewanego.  
2. **Infiltracja:** Automatyczny dodatek na nieszczelności przegród zewnętrznych, wyliczany proporcjonalnie do sumarycznej długości i powierzchni wyodrębnionych krawędzi EXTERIOR.

---

### **6.5. Agregacja i Zapis Wyników (Output Data)**

Proces obliczeniowy jest uruchamiany asynchronicznie (np. poprzez Web Workers, by nie blokować wątku UI React). Po zakończeniu cyklu, każda strefa w useZoneStore zostaje wzbogacona o obiekt ThermalBalance:

TypeScript

interface ThermalBalance {

  winterHeatingLoad: number; // \[W\] Szczytowe zapotrzebowanie na ciepło (zima)

  summerCoolingLoad: {

    sensible: number; // \[W\] Ciepło jawne (do zbijania temperatury)

    latent: number;   // \[W\] Ciepło utajone (odpowiada za wilgotność)

    total: number;    // \[W\] Suma zapotrzebowania chłodniczego

  };

  metrics: {

    loadPerSquareMeter: number; // \[W/m2\] Wskaźnik referencyjny

    solarGainPercentage: number; // \[%\] Udział zysków przez okna

  };

}

Uzyskane w tym module wyniki stają się bezpośrednim fundamentem do automatycznego wymiarowania urządzeń (np. doboru przepływów dla klimakonwektorów FCU, belek chłodzących czy kratek nawiewnych w module sieciowym WENTCAD).

Oto szczegółowe rozwinięcie siódmego, ostatniego rozdziału. Zostało ono zredagowane jako rygorystyczny "Prompt/Master-Plan" (tzw. Meta-Prompt) przeznaczony bezpośrednio dla agentów programistycznych AI (takich jak Cursor, GitHub Copilot czy ja w kolejnych sesjach). Gwarantuje to, że AI nie pogubi się w złożoności matematycznej i wdroży moduł WATT w logicznych, bezpiecznych iteracjach.
